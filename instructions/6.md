Next step: add Home page / task workspace UI.

Current state:
- Backend task APIs are implemented under /api/tasks
- Upload flow already creates a task and uploads dataset
- Data persists under DATA_DIR + SQLite
- Frontend currently still behaves like a single workbench and creates task on upload

Goal:
Make the app task-based from the user perspective.

Implement:

1. Home page
- Show all existing tasks from GET /api/tasks
- Show task cards:
  - task_name
  - dataset_name
  - total_rows
  - num_stages
  - updated_at
- Buttons:
  - New Task
  - Open
  - Rename
  - Delete
  - Refresh

2. New task flow
- User clicks New Task
- Ask for task name
- POST /api/tasks
- Open that task workspace
- If task has no dataset, show upload panel

3. Open existing task
- Set current taskId
- Load GET /api/tasks/{task_id}/stages
- Select current_stage_id if available, otherwise latest stage
- Restore:
  - stage timeline
  - kept rows
  - removed rows
  - charts
  - stats

4. Task page
- Add Back to Home button
- Show current task name at top
- If no dataset/stages, show upload prompt
- Existing workbench UI should be scoped to taskId

5. Upload behavior
- If current task has no dataset:
  upload into that task
- Do NOT automatically create a new task on upload anymore unless user is on Home and explicitly chooses New Task
- If task already has dataset and user tries upload again:
  show confirmation that this will reset/replace that task’s stages, or disable for now

6. Rename/delete
- Rename calls PATCH /api/tasks/{task_id}
- Delete calls DELETE /api/tasks/{task_id}, then returns to Home
- Ask confirmation before delete

7. UI style
Keep current Apple-style black/white minimal design.

8. Persistence expectation
After refresh:
- app should load Home
- existing tasks should still be visible
- opening a task should recover its saved stages from backend

9. Do not change backend unless necessary.
Only add missing small endpoints if frontend needs them.
