Revise the platform design based on the real input data.

The actual dataset mainly contains:

* question
* response

It usually does NOT contain:

* correctness
* compiled
* runtime_ms
* behavior_type
* source_model

Therefore, remove generic metadata-based filters from the first version.

Do NOT implement:

* correctness_only
* compiled_only
* runtime_range_filter
* remove_slow_samples
* behavior_filter
* source_model_filter

These are not useful for the current dataset.

# Real filters to implement

## 1. HackingFilter

Use the provided hack detector script.

Map:

* question -> instruction
* response -> output

The filter should detect:

* ATen / Torch fallback
* cuBLAS / cuDNN fallback
* no custom **global** kernel
* ModelNew.forward torch fallback
* required op fallback

The user can choose filter level:

* 1 loose
* 2 medium
* 3 strict
* 4 very strict

Removed rows must include:

* removal_reason
* severity
* hack_types
* forward_fallback_ops
* cuda_fallback_ops
* has_global_kernel

---

## 2. DedupFilter

Use the provided dedup script as reference.

Support:

* exact duplicate by question
* exact duplicate by response
* exact duplicate by question + response
* optional near-duplicate mode if easy

Removed rows must include:

* removal_reason
* duplicate_key
* kept_row_id if available

---

## 3. FormatValidityFilter

Check whether response looks like a valid CUDA SFT answer.

Config options:

* require_modelnew
* require_load_inline
* require_global_kernel
* require_forward
* require_cuda_source

Examples:

* missing class ModelNew
* missing **global**
* missing load_inline
* empty response
* empty question

Removed rows must include exact reason.

---

## 4. LengthAnomalyFilter

Support:

* min_question_chars
* max_question_chars
* min_response_chars
* max_response_chars

Also detect obvious truncation:

* unfinished code fence
* response ends inside a code block
* incomplete Python/CUDA structure if simple to detect

Removed rows must include exact reason.

---

## 5. SignatureExtraction

This is not primarily a deletion filter.

Extract signature-like metadata from question:

* operator family
* optimization stage
* optimization technique
* fused operator description if present

Use this extracted signature for:

* distribution visualization
* drill-down view
* subset filtering

Save extracted fields in the dataset table:

* signature
* operator_family
* stage
* technique

Do not hard-code too many operator rules. Use robust regex extraction from the question text where possible.

# Required interaction model

Keep the following platform features:

## Batch apply filters

Users can select multiple filters at once:

* hacking
* dedup
* format validity
* length anomaly

Click:
Apply selected filters

This creates ONE new stage.

The stage records:

* selected filters
* configs
* input_count
* output_count
* removed_count
* per_filter_removed_count
* removed rows with all reasons

If a row is removed by multiple filters, keep all reasons.

---

## Signature drill-down

Users can click a signature/operator family in the distribution chart.

This creates a temporary view only, not a new stage.

The user sees:

* subset rows
* subset stats
* subset distributions

Then the user may apply selected filters only to this subset.

When filters are applied to a subset:

* create a new stage
* only mutate the selected subset
* merge kept subset rows back with untouched non-subset rows
* record the view_filter in the stage metadata

Example:
Stage has 10,000 rows.
User views signature=conv_fused, 500 rows.
User applies hacking filter to this subset.
80 rows are removed.
New stage has 9,920 rows.
The other 9,500 rows are unchanged.
