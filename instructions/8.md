Update the workbench around subset-based curation.

Current state:
- Task-based persistence exists.
- Signature subset selection already exists.
- Filters exist but need to be refocused for our real question/response SFT data.

Goal:
Users should be able to select a subset by data dimensions, apply filters only to that subset, and then either export the full updated dataset or export only the selected subset.

---

# 1. Subset selector above data table

Add a dedicated subset selector above the main data table.

It should support selecting subset by:

- signature
- operator_family
- stage
- technique

The current signature subset behavior should be generalized to all these fields.

Example:
User selects:
operator_family = conv
stage = 3
technique = shared_memory

Then the table, stats, and charts should show only matching rows from the current stage.

This subset selection is a view only until filters are applied.

---

# 2. Apply filters to selected subset

If a subset is active and the user applies filters:

- only apply filters to rows in the subset
- keep all non-subset rows unchanged
- create a new stage
- save the new full-stage dataset

Example:
Current stage has 10,000 rows.
Subset has 1,000 rows.
Filter removes 200 rows from subset.
New full stage has 9,800 rows.
The other 9,000 rows are unchanged.

Stage metadata should record:

- subset_filter
- affected_count
- untouched_count
- removed_count
- selected filters
- filter configs

---

# 3. Export options

Add export options:

1. Export full current stage
- includes all rows after subset modifications

2. Export current subset only
- only exports rows matching active subset selector

Support JSONL first.
CSV optional.

---

# 4. Add distribution balancing filters

Add filters useful for balancing data distribution.

## RandomDropFilter

Purpose:
Randomly remove a percentage of rows from current selection.

Config:
- drop_fraction: float between 0 and 1
- random_seed: int

Example:
drop 20% of selected subset.

Removed rows should include:
- removal_reason = "random_drop"
- drop_fraction
- random_seed

---

## BalanceToMeanFilter

Purpose:
Balance data across a selected category.

Config:
- group_by: one of signature, operator_family, stage, technique
- target: "mean"
- random_seed

Behavior:
- compute counts per group within current stage or current subset
- compute mean count
- for groups above mean, randomly remove rows until count = floor(mean)
- groups below mean remain unchanged

Example:
signature counts:
A = 100
B = 40
C = 10
mean = 50
Remove 50 from A.
B and C unchanged.

Removed rows should include:
- removal_reason = "balance_to_mean"
- group_by
- group_value
- original_group_count
- target_count

Important:
This filter is useful for distribution balancing, not quality filtering.

# 6. PromptValidityFilter

Check question quality.

Config:
- require_original_pytorch_operator
- require_task_instruction
- require_input_description
- require_stage_plan

Possible removal reasons:
- missing Original PyTorch Operator section
- missing task instruction
- missing input/get_inputs description
- missing current stage plan

---

# 7. ImplementationCompletenessFilter

Check response implementation completeness.

Config:
- require_modelnew
- require_forward
- require_load_inline
- require_cuda_source
- require_global_kernel
- require_custom_kernel_call

Possible removal reasons:
- missing class ModelNew
- missing forward method
- missing load_inline
- missing CUDA source string
- missing __global__ kernel
- forward does not call fused/custom CUDA op

This is different from HackingFilter:
- HackingFilter detects cheating/fallback.
- ImplementationCompletenessFilter detects incomplete/non-code/invalid answer structure.

# 9. Backend API

Extend existing view endpoint to support multiple filters:

GET /api/tasks/{task_id}/stages/{stage_id}/view

Query params:
- signature
- operator_family
- stage
- technique
- limit
- offset

Apply filters endpoint should accept:

{
  "base_stage_id": "...",
  "subset_filter": {
    "signature": "...",
    "operator_family": "...",
    "stage": "...",
    "technique": "..."
  },
  "filters": [...]

GET /api/tasks/{task_id}/export?stage_id=...&subset_only=true&signature=...&stage=...

# 10. Important behavior

Subset selection itself does not create a stage.
Only applying filters creates a new stage.

All removed rows must keep exact removal reasons.

All stage outputs must remain recoverable from storage.
