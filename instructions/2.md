Update the product design:

The platform must support two different concepts:

1. Filtering operation
2. Data exploration / drill-down view

---

# 1. Batch filter application

Users must be able to select multiple filters at once.

Example:
- remove_hacking
- remove_duplicates
- correctness_only
- compiled_only
- runtime_range_filter

Then click:
"Apply selected filters"

This creates ONE new stage, not one stage per filter.

The new stage should record:
- selected filters
- config for each filter
- input_count
- output_count
- total_removed_count
- removed rows
- removal_reason per row
- per_filter_removed_count

If a row is removed by multiple filters, store all reasons:
removal_reasons = ["hacked == true", "duplicate_of_row_123"]

The UI should show:
- total removed
- removed by each filter
- before/after distributions
- removed rows table with all reasons

---

# 2. One-click select all filters

Filter panel should include:
- Select all base filters
- Select all quality filters
- Select all structure filters
- Select all distribution filters
- Clear all

User can quickly enable common production presets.

Add presets:
- Basic cleanup: remove_hacking + remove_duplicates + remove_empty_prompt + remove_empty_response
- Quality only: compiled_only + correctness_only + runtime_range_filter
- Full cleanup: basic cleanup + quality only

---

# 3. Exploration / drill-down view

Distribution charts should be interactive.

When user clicks a category in a chart, such as:
- signature = X
- problem_type = Y
- source_model = Z
- behavior_type = B

Do NOT create a new stage immediately.

Instead, create a temporary view filter:
current_view = subset of selected stage

The user should see:
- subset table
- subset statistics
- subset distributions
- clear view filter button

This is only exploration, not data mutation.

---

# 4. Apply filters to subset

If user is in a drill-down subset view, they can still apply selected filters.

When applying filters from a subset view:

Option A:
Apply to subset only, then merge kept subset back with untouched non-subset rows.
This creates a new stage.

The stage should record:
- base_stage_id
- view_filter used
- selected filters
- affected_row_count
- removed_count
- untouched_row_count

Example:
Base stage has 10,000 rows.
User drills down to signature=conv_2d, 500 rows.
User applies remove_duplicates to subset.
If 80 rows removed:
New stage has 9,920 rows.
The other 9,500 rows are unchanged.

---

# 5. UI changes

Left panel:
- filter groups with checkboxes
- select all / clear all
- preset buttons
- apply selected filters button

Top:
- pipeline timeline showing stages

Center:
- current table
- if drill-down active, show banner:
  "Viewing subset: signature = conv_2d"

Right:
- clickable distribution charts

Bottom:
- removed rows table
- per-filter removal summary

---

# 6. Backend API update

POST /datasets/{dataset_id}/apply-filters

Request:
{
  "base_stage_id": "...",
  "view_filter": {
    "field": "signature",
    "value": "conv_2d"
  } | null,
  "filters": [
    {
      "filter_type": "remove_hacking",
      "filter_config": {}
    },
    {
      "filter_type": "remove_duplicates",
      "filter_config": {}
    }
  ]
}

Return:
{
  "new_stage_id": "...",
  "input_count": ...,
  "output_count": ...,
  "removed_count": ...,
  "per_filter_removed_count": {...}
}

GET /datasets/{dataset_id}/stages/{stage_id}/view
Query params:
- field
- value

Returns subset rows + subset stats.

---

# 7. Important

Keep the stage concept.
But stages should represent meaningful user actions, not necessarily single filters.

A stage can be:
- one filter
- multiple filters
- filters applied to a subset

Every stage must remain auditable.
