Extend the existing backend to support **task-based persistence and shared storage**.

Do NOT rewrite the whole app. Build on top of current `/api` structure.

---

# Goal

Multiple users (colleagues) should be able to:

* create tasks
* upload datasets per task
* apply filters
* leave and come back later
* see all tasks on a home page

Data must persist across server restarts.

---

# 1. Introduce Task concept

Add a new top-level entity: `Task`

Each task represents one dataset curation workflow.

Fields:

```text
task_id
task_name
created_at
updated_at
dataset_name
current_stage_id
total_rows
num_stages
```

---

# 2. Introduce Stage persistence

Each stage must be saved and recoverable.

Fields:

```text
stage_id
task_id
stage_index
base_stage_id
filter_config_json
view_filter_json
input_count
output_count
removed_count
affected_count
untouched_count
kept_file_path
removed_file_path
created_at
```

---

# 3. Storage (IMPORTANT)

Implement persistent storage with **two layers**:

## (A) Metadata → SQLite (default)

Use SQLite file:

```text
data/tasks.db
```

Store:

* tasks
* stages

If `DATABASE_URL` is provided, allow switching to Postgres later, but for now prioritize SQLite.

---

## (B) Dataset files → Local filesystem (for now)

Store all dataset files under:

```text
data/tasks/{task_id}/
```

Structure:

```text
raw/input.jsonl

stages/stage_0/kept.jsonl
stages/stage_0/removed.jsonl

stages/stage_1/kept.jsonl
stages/stage_1/removed.jsonl
```

Each stage must write:

* kept rows
* removed rows

---

## IMPORTANT

Do NOT keep dataset only in memory.

All stages must be reloadable from disk after restart.

---

# 4. API changes

Keep `/api` prefix.

Add:

## Task APIs

```text
POST /api/tasks
GET /api/tasks
GET /api/tasks/{task_id}
DELETE /api/tasks/{task_id}
PATCH /api/tasks/{task_id}
```

---

## Dataset upload (per task)

```text
POST /api/tasks/{task_id}/datasets/upload
```

This should:

* save raw dataset to disk
* create stage_0 (kept = full dataset)
* insert stage_0 into DB

---

## Stage APIs

```text
GET /api/tasks/{task_id}/stages
GET /api/tasks/{task_id}/stages/{stage_id}
GET /api/tasks/{task_id}/stages/{stage_id}/rows
GET /api/tasks/{task_id}/stages/{stage_id}/removed-rows
```

---

## Apply filters (modify existing endpoint)

Change to:

```text
POST /api/tasks/{task_id}/apply-filters
```

Behavior:

* load base stage kept.jsonl
* apply filters
* write new stage files
* insert new stage in DB
* update task.current_stage_id

---

# 5. Existing filters

Keep current filters:

* hacking
* dedup
* format
* length

No need to add correctness/runtime filters.

---

# 6. Startup recovery

On server start:

* load all tasks from DB
* stages are NOT loaded into memory
* files are read on demand from disk

---
